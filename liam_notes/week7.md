# Week 7: information architecture, visual hierarchy and validation

## Class focus

The Week 7 material focuses on information architecture, visual hierarchy, colour and typography. It also asks teams to prepare a first-sprint prototype for testing and to check that acceptance criteria, navigation and flow make sense.

## Research inputs for information architecture

Information architecture creates the structure of a website, interface or application so users can understand where they are, where information is located and how to reach it. It must consider:

- users;
- context;
- content.

The structure should be informed by research artefacts such as personas, scenarios, journey maps and usability tests. It also requires an inventory of the content and functions that the structure must support.

The lecture distinguishes:

- **User flow**: how a user experiences the system over time.
- **Information architecture**: how content is structured and presented at each touchpoint.

Information architecture can be considered through:

- **Ontology**: understanding what the content and concepts mean;
- **Taxonomy**: organising and naming the content;
- **Choreography**: applying that structure intentionally through the experience.

Context, people and culture can change all three.

## Requirements and structure analysis

The slide process for defining information architecture is:

1. Define user goals clearly and consider organisational goals.
2. Create and audit a content inventory.
3. Group information, potentially through card sorting.
4. Develop a common taxonomy.
5. Define navigation, using flowcharts where useful.
6. Test the structure.

Priority should be given to discoverability over novelty and usability over aesthetics.

Relevant architecture principles include objects, choices, disclosure, exemplars, front doors, multiple classification, focused navigation and growth. Progressive disclosure is important because information shown before a user is ready to process it becomes noise.

## Prototype validation notes

Before testing a first-sprint prototype, check that:

- acceptance criteria describe observable behaviour;
- the navigation supports the intended user flow;
- content is grouped and labelled consistently;
- users can recognise where they are and what to do next;
- important information is discoverable without excessive choices;
- error and recovery paths are represented;
- accessibility has been considered.

## Visual hierarchy notes

Visual hierarchy controls the order in which users process information. It can be created through:

- size and scale;
- colour and contrast;
- typography;
- spacing and proximity;
- alignment;
- repetition;
- negative space;
- focal points and balance.

Related elements should be grouped through proximity and alignment. Repeated visual patterns help users learn the interface. Larger, higher-contrast or isolated elements appear more prominent, so emphasis should match the importance of the content.

## Colour and typography checks

- Colour choices should consider emotion, harmony, contrast, scale and context.
- Colour meaning may change across cultures.
- Colour-blind users must be considered, and colour choices should be tested with users.
- Information should not rely on colour alone.
- Typography should prioritise readability, legibility and accessibility.
- Too many fonts reduce consistency.
- The slides recommend experimenting with line height around 130–180% and limiting line length to roughly 70–80 characters.
- The slides identify a base text size around 16 as a useful minimum, depending on device and font context.
- Text contrast should meet a 4.5:1 ratio where applicable.
- Text should remain readable and functional when scaled to 200%.
- Images of text should be avoided unless necessary, such as for a logo.

## Weekly record template

| Item | Note to record |
| --- | --- |
| User goal | What is the user trying to locate or complete? |
| Context | Where and under what conditions does the task occur? |
| Content inventory | Which information and functions must be supported? |
| Taxonomy | How are items grouped and labelled? |
| User flow | What are the main steps and decision points? |
| Acceptance criteria | Which observable behaviours validate the flow? |
| Visual hierarchy | What should users notice first, second and third? |
| Accessibility | How are contrast, typography, scaling and non-colour cues handled? |
| Test question | What part of the structure or presentation still needs user validation? |
